# Mellow — AI Therapy Voice Assistant

> A real-time, voice-first AI therapist powered by Pipecat, NVIDIA NIM, Deepgram, and ElevenLabs — with Cekura evaluating every session automatically.


---

## 1. What is this?

Mellow is a voice-based mental health support app that lets users have full-duplex, real-time conversations with an AI therapist — no typing required. Users speak naturally; the AI listens, thinks, and responds in a warm, human-sounding voice.

Key capabilities:

- **Full-duplex voice conversation** via WebRTC — the AI speaks back in real time using ElevenLabs TTS and Pipecat hosted on Pipecat cloud using Krisp
- **Onboarding flow** that personalizes the system prompt to each user's mood, topic, and preferences
- **Crisis detection** — the LLM can call a `detect_crisis_intent` tool mid-conversation; the client immediately shows an intervention modal with emergency resources
- **Automatic session summaries** generated by NVIDIA NIM (Llama 3.1 8B) and saved to the user's dashboard
- **Cekura evaluation** — every session transcript is automatically sent to Cekura after the call ends for scoring and observability
- **Versioned system prompts** — the bot loads the latest `v*.txt` prompt file so prompt iteration requires zero redeploy
- **Message persistence** — all user and AI turns are written to PostgreSQL in real time
- **JWT authentication**, session history, and a personal dashboard with session summaries

## 3. How I used Pipecat, NVIDIA NIM, and Cekura

### Pipecat — the voice pipeline backbone

The entire real-time voice experience runs on a Pipecat Python server (`backend fast api/pipecat-quickstart/pipecat-quickstart/server/bot.py`). The pipeline is:

```
Browser mic
  → SmallWebRTC transport
  -> Krisp Pipecat cloud
  → Deepgram STT
  → Silero VAD + RNNoise audio filter
  → LLM (NVIDIA NIM via OpenAI-compatible API)
  → ConversationPersister (custom FrameProcessor)
  → ElevenLabs TTS (eleven_flash_v2_5)
  → WebRTC transport back to browser
```

The frontend uses `@pipecat-ai/client-js` with `SmallWebRTCTransport` to connect directly to the bot server at `/start`. Session identity (`sessionId`, `userId`, personalized `systemPrompt`) is forwarded in the request body so the bot can associate its messages with the correct DB record.

A custom `ConversationPersister` frame processor sits between the LLM and TTS stages. It watches for `LLMFullResponseEndFrame` events, snapshots the shared `LLMContext`, and persists every new user/assistant message to PostgreSQL in real time — all without blocking the audio pipeline.

Pipecat Cloud deploy config (`pcc-deploy.toml` + `Dockerfile`) is also included for one-command cloud deployment.

### NVIDIA NIM — LLM for voice conversation and session summaries

NVIDIA NIM powers **two distinct inference calls** in MindStudio:

**1. Real-time voice LLM (in the Pipecat bot)**

```python
llm = OpenAILLMService(
    api_key=os.getenv("NVIDIA_API_KEY"),
    base_url="https://integrate.api.nvidia.com/v1",
    settings=OpenAILLMService.Settings(
        model="meta/llama-3.1-8b-instruct",
        extra={"stop": ["<think>", "\n<think>"]},  # suppress chain-of-thought leaks
    ),
)
```

The `stop` token for `<think>` was a critical addition — without it, Llama's chain-of-thought reasoning leaked into spoken responses, producing nonsensical audio output.

**2. Post-session summarization (Next.js API route)**

After each session ends, the frontend calls `/api/summarize` which sends the full transcript to NVIDIA NIM:

```typescript
const response = await fetch("https://integrate.api.nvidia.com/v1/chat/completions", {
  body: JSON.stringify({
    model: "meta/llama-3.1-8b-instruct",
    temperature: 0.5,
    max_tokens: 160,
    messages: [
      { role: "system", content: "You summarize a therapy session for the client to read afterward. Write 2-3 warm, plain, non-clinical sentences..." },
      { role: "user", content: `Summarize this session:\n\n${conversation}` },
    ],
  }),
});
```

The summary is saved to localStorage and displayed on the user's dashboard.

### Cekura — automated transcript evaluation

Every session transcript is automatically evaluated by Cekura (`backend fast api/.../server/cekura.py` and `backend/src/Service/CekuraService.ts`). On WebRTC disconnect, the bot:

1. Flushes any buffered messages via `ConversationPersister.flush()`
2. Fetches the complete session transcript from PostgreSQL
3. Saves it locally to `server/transcripts/<session_id>.json`
4. POSTs it to Cekura's observability API:

```python
payload = {
    "agent_id": agent_id,          # "therapist-bot"
    "run_id": session_id,          # UUID linking to DB session
    "transcript_type": "pipecat",
    "transcript_json": messages,   # [{role, content, timestamp}]
}
await client.post(CEKURA_API_URL, json=payload, headers={"X-CEKURA-API-KEY": api_key})
```

**What we were trying to accomplish with Cekura:**

We wanted to measure and improve the quality of the AI's therapeutic responses across different prompt versions. Specifically:
- Is the AI responding empathetically without being scripted or robotic?
- Does it ask follow-up questions naturally, or does it over-explain?
- Is it appropriately cautious with crisis signals (not triggering the crisis tool on ordinary stress)?

We iterated through three system prompt versions (`prompts/v1.txt` → `v2.txt` → `v3.txt`), using Cekura's session scoring to compare. The bot always loads the latest `v*.txt` file, so we could test new prompts instantly. Cekura's observability scores helped us identify that early prompt versions caused the AI to:
- Summarize emotions back to the user too repetitively
- End almost every turn with a question (felt interrogative, not conversational)
- Occasionally trigger `detect_crisis_intent` incorrectly on mild expressions of worry

Iterating on the system prompt with Cekura feedback noticeably improved response naturalness and reduced false crisis triggers.

## 5. Feedback on the tools

### NVIDIA NIM / Meta Llama 3.1 8B Instruct

**What worked well:**
- The OpenAI-compatible API made integration with Pipecat's `OpenAILLMService` completely seamless — one `base_url` swap and it just worked
- Latency for the 8B model was fast enough for real-time voice conversation; responses felt natural in the Pipecat pipeline
- Function/tool calling worked correctly on first try for the crisis detection use case
- The summarization quality at `temperature=0.5` was consistently warm and non-clinical — exactly what we needed for the post-session card

**What could be better:**
- **Chain-of-thought leakage** was the biggest issue: without `stop: ["<think>"]`, the model's internal reasoning text flowed into the spoken response. This is a significant UX bug for voice applications, and ideally the API would strip `<think>...</think>` blocks server-side before returning tokens
- The 8B model occasionally lost the "human-sounding therapist" persona mid-session, reverting to a more formal, listy response style. A Nemotron-based model or a fine-tuned therapy model might hold the persona more consistently
- No built-in streaming stop-sequence enforcement — we had to rely on the `stop` array in extra params, which only works at the token level (doesn't handle partial `<think>` already in the buffer)
- We'd love to try **Nemotron** models — the 70B parameter range might hold the conversational persona much better and handle more nuanced emotional cues without reverting to clinical language

### Cekura

**What worked well:**
- The `transcript_type: "pipecat"` mode was a perfect fit — our transcript format from `ConversationPersister` mapped exactly to what Cekura expects
- Scoring feedback was actionable: we could clearly see which turns felt robotic vs. natural, and that drove our prompt iterations
- The observability API is simple and reliable — we never had a failed POST to Cekura in testing

**Potential improvements / bugs found:**
- There's no webhook or polling endpoint to retrieve scores programmatically after submission. We had to check the Cekura dashboard manually. A `/results/{run_id}` GET endpoint would let us close the loop automatically (e.g. log the score to our DB and display it in the user dashboard)
- When sending a transcript with only 1–2 turns (very short sessions), the API accepted it but scoring felt less meaningful. A minimum-length recommendation in the docs would help
- We'd love to see diff-style comparison between `run_id`s — e.g., "here's how prompt v2 scored vs v1 across 10 sessions" — this would make the self-improvement loop much tighter

## Running Locally

### Prerequisites

- Node.js 20+
- Python 3.11+ with [uv](https://docs.astral.sh/uv/)
- PostgreSQL database (Supabase free tier works)
- API keys: NVIDIA, Deepgram, ElevenLabs, Cekura

### 1. Clone the repo

```bash
git clone <repo-url>
cd AI-Therapist
```

### 2. Start the Pipecat bot server

```bash
cd "backend fast api/pipecat-quickstart/pipecat-quickstart/server"
cp .env.example .env
# Fill in: NVIDIA_API_KEY, DEEPGRAM_API_KEY, ELEVENLABS_API_KEY,
#           ELEVENLABS_VOICE_ID, DATABASE_URL, CEKURA_API_KEY, CEKURA_AGENT_ID
uv sync
uv run bot.py
```

Bot runs at **http://localhost:7860**

### 3. Start the frontend

```bash
cd frontend
cp .env.example .env.local
# Set: NEXT_PUBLIC_API_URL, NEXT_PUBLIC_WS_URL, NVIDIA_API_KEY
npm install
npm run dev
```

Frontend runs at **http://localhost:3000**

### 4. Use the app

Go to http://localhost:3000 → Register → complete Onboarding → Start Session → speak.
